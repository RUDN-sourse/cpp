
# Техническое задание на разработку банковской системы

## 1. Общие положения

### 1.1. Наименование проекта
**"Упрощенная банковская система"** - консольное приложение для управления банковскими операциями.

### 1.2. Цель разработки
Создание объектно-ориентированного приложения, демонстрирующего практические навыки работы со структурами данных, классами и организацией проекта на C++.

### 1.3. Технологический стек
- Язык программирования: C++ (стандарт C++17 или выше)
- Система сборки: Makefile или CMake
- ОС: кроссплатформенное решение (Windows/Linux/macOS)

## 2. Требования к архитектуре проекта

### 2.1. Структура директорий
Проект должен быть организован в следующей иерархии:
```
BankSystem/
├── include/          // Заголовочные файлы (.h)
├── src/             // Файлы реализации (.cpp)
├── docs/            // Документация
└── build/           // Сборка (опционально)
```

### 2.2. Модульная структура
Приложение должно состоять из следующих модулей:
- Модуль управления клиентами
- Модуль управления счетами  
- Модуль транзакций
- Модуль банковских операций
- Главный исполнительный модуль

## 3. Функциональные требования

### 3.1. Управление клиентами
**Требуется реализовать:**
- Добавление нового клиента в систему
- Удаление клиента (при отсутствии активных счетов)
- Поиск клиента по идентификатору
- Просмотр информации о всех клиентах
- Редактирование данных клиента
- Поддержка двух типов клиентов: обычный и премиум

### 3.2. Управление банковскими счетами
**Требуется реализовать:**
- Открытие новых счетов (расчетный, сберегательный)
- Закрытие счетов (при нулевом балансе)
- Привязка счетов к клиентам
- Просмотр всех счетов конкретного клиента
- Поиск счета по номеру
- Проверка статуса счета (активный/закрытый)

### 3.3. Банковские операции
**Требуется реализовать:**
- Внесение денежных средств на счет
- Снятие денежных средств со счета
- Переводы между счетами
- Проверка достаточности средств при операциях
- Ведение истории всех транзакций

### 3.4. Отчетность и аналитика
**Требуется реализовать:**
- Отображение общей информации о банке
- Список всех зарегистрированных клиентов
- Список всех открытых счетов
- Историю проведенных транзакций
- Статистику по активным клиентам и счетам

## 4. Требования к структурам данных

### 4.1. Структура "Адрес"
Должна содержать следующие поля:
- Улица и дом
- Город
- Почтовый индекс
- Страна

### 4.2. Структура "Дата"
Должна содержать следующие поля:
- День
- Месяц  
- Год

## 5. Требования к классам (ООП)

### 5.1. Базовый класс "Клиент"
**Обязательные атрибуты:**
- Уникальный идентификатор клиента
- Имя и фамилия
- Адрес (структура)
- Дата регистрации
- Статический счетчик клиентов

**Обязательные методы:**
- Конструкторы (по умолчанию, параметризованный)
- Виртуальный деструктор
- Геттеры для всех атрибутов
- Сеттеры для изменяемых атрибутов
- Виртуальные методы отображения информации
- Статические методы для работы со счетчиком

### 5.2. Производный класс "Премиум-клиент"
**Дополнительные атрибуты:**
- Уровень премиум-статуса
- Процент скидки

**Требования:**
- Наследование от базового класса Клиент
- Переопределение виртуальных методов
- Дополнительные методы, специфичные для премиум-клиентов

### 5.3. Класс "Банковский счет"
**Обязательные атрибуты:**
- Уникальный номер счета
- Идентификатор владельца-клиента
- Текущий баланс
- Тип счета
- Дата открытия
- Статус активности
- Статический счетчик счетов

**Обязательные методы:**
- Операции пополнения, снятия, перевода
- Методы активации/деактивации счета
- Статические методы генерации номеров счетов
- Методы получения информации о счете

### 5.4. Класс "Транзакция"
**Обязательные атрибуты:**
- Уникальный идентификатор транзакции
- Счет-отправитель
- Счет-получатель
- Сумма операции
- Дата и время транзакции
- Тип операции
- Статический счетчик транзакций

### 5.5. Класс "Банк" (управляющий)
**Обязательная функциональность:**
- Управление коллекциями клиентов, счетов и транзакций
- Реализация бизнес-логики банковских операций
- Поисковые функции по клиентам и счетам
- Генерация отчетов
- Приватные методы для внутренней логики

## 6. Требования к принципам ООП

### 6.1. Инкапсуляция
- Использование модификаторов доступа private/protected
- Предоставление публичных геттеров/сеттеров
- Сокрытие внутренней реализации

### 6.2. Наследование
- Создание иерархии классов Клиент → Премиум-клиент
- Правильное использование модификаторов наследования

### 6.3. Полиморфизм
- Применение виртуальных методов
- Переопределение методов в производных классах
- Использование указателей на базовый класс

### 6.4. Композиция и агрегация
- Правильное отношение между Bank и Client/Account
- Использование контейнеров STL для хранения объектов

## 7. Требования к обработке ошибок

### 7.1. Валидация данных
- Проверка корректности сумм операций
- Валидация идентификаторов клиентов и счетов
- Проверка существования объектов при операциях
- Контроль достаточности средств

### 7.2. Обработка исключительных ситуаций
- Корректная реакция на некорректный ввод
- Защита от операций с несуществующими счетами
- Обработка попыток нарушения бизнес-правил

## 8. Требования к пользовательскому интерфейсу

### 8.1. Консольное меню
- Интуитивно понятная навигация
- Четкие инструкции для пользователя
- Обработка пользовательского ввода
- Возможность выхода из программы

### 8.2. Форматы вывода
- Читаемое представление данных
- Форматирование числовых значений (балансы)
- Структурированное отображение списков

## 9. Технические требования

### 9.1. Качество кода
- Соблюдение naming conventions
- Отсутствие магических чисел
- Комментарии для сложных алгоритмов
- Логичное разделение на методы

### 9.2. Управление памятью
- Отсутствие утечек памяти
- Корректная работа с динамической памятью
- Правильное использование умных указателей (опционально)

### 9.3. Статические элементы
- Использование статических полей для счетчиков
